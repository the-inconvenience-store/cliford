# Implementation Plan: Complete Generated App Wiring

**Branch**: `002-002-complete-generated` | **Date**: 2026-04-11 | **Spec**: [spec.md](spec.md)

## Summary

Wire all 26 identified feature gaps, stubs, and improvements into the Cliford code generator's existing pipeline. The majority of underlying generators already exist in `internal/sdk/`, `internal/cli/auth.go`, `internal/hooks/`, and `internal/distribution/`; this work completes their implementation and connects them through the generation pipeline. The result is a generated Go app that authenticates, retries, paginates, prompts interactively, confirms destructive actions, renders table output, and supports before/after request hooks — without any manual post-generation edits.

## Technical Context

**Language/Version**: Go 1.22+  
**Primary Dependencies**: Cobra, Bubbletea, Bubbles, Lipgloss, Viper, kin-openapi, oapi-codegen, zalando/go-keyring (new), golang.org/x/oauth2 (new), hashicorp/go-plugin (new)  
**Storage**: OS keychain (zalando/go-keyring) → encrypted file → YAML/TOML config (Viper)  
**Testing**: `go test ./...`, golden file tests, integration test generating from reference spec  
**Target Platform**: Linux, macOS, Windows (cross-platform via GoReleaser)  
**Project Type**: CLI code generator (generates CLI + TUI apps)  
**Performance Goals**: Generation completes in <10s for typical specs; generated app adds <5ms overhead per request for auth/retry middleware  
**Constraints**: Generated code MUST pass `go build`, `go vet`, `go test` immediately with no manual steps  
**Scale/Scope**: Support OpenAPI specs with 1–200 operations; single developer through enterprise use

## Constitution Check

*GATE: Must pass before implementation. Re-checked after Phase 1 design.*

| Principle | Check | Status |
|-----------|-------|--------|
| I. OpenAPI as Source of Truth | All new behavior (auth, retry, pagination, hooks) derives from spec metadata + Cliford config; `x-cliford-display` used for table column selection | ✅ PASS |
| II. SDK-First Architecture | Generated CLI/TUI MUST invoke SDK methods, not make direct HTTP calls. Current generated code makes direct HTTP calls — this feature must fix that by routing through the generated SDK layer | ⚠️ MUST FIX — see implementation note below |
| III. Three Modes, One Codebase | Auth, retry, and hooks shared across CLI/TUI/headless adapters via shared `internal/client/factory.go` | ✅ PASS |
| IV. Configuration Over Convention | Viper config; credential resolution order `flags > env > keychain > encrypted file > config file` enforced; retry params overridable at runtime via FeaturesConfig | ✅ PASS |
| V. Custom Code Survives Regeneration | Not addressed in this feature; `cliford diff` exists; custom regions out of scope | ➡️ DEFERRED (existing feature) |
| VI. Extensibility via Hooks | Shell + go-plugin hooks implemented; not required for basic operation | ✅ PASS |
| VII. Idiomatic Go Output | All generated code must pass `go vet`, `gofmt`; dependencies minimized | ✅ PASS |
| VIII. Security by Default | zalando/go-keyring as default; encrypted-file fallback; plain-text config as last resort; all sensitive headers redacted | ✅ PASS (after spec correction) |

**Principle II Implementation Note**: The existing `internal/cli/generator.go` generates `RunE` functions that build and execute HTTP requests directly. This violates Constitution Principle II. The fix is: generated `RunE` functions MUST call SDK methods generated by oapi-codegen rather than constructing HTTP requests inline. The `http.Client` with auth/retry transport is constructed once in `main.go` and injected into the SDK client. CLI commands call SDK methods; SDK client uses the injected `http.Client`. This is the primary architectural change required.

## Implementation Phases

### Phase A: Foundation (Constitution Compliance)

**Goal**: Establish SDK-First architecture and credential resolution. Everything else builds on this.

**A1 — SDK-First Refactor**
- `internal/cli/generator.go`: Replace inline HTTP request construction in `RunE` with calls to oapi-codegen-generated SDK client methods
- `internal/sdk/auth_transport.go`: Generate `AuthTransport` that implements `http.RoundTripper`, reads resolved credentials, injects correct headers per scheme type
- Generated `internal/client/factory.go`: Assemble `http.Client{Transport: AuthTransport{Base: RetryTransport{Base: http.DefaultTransport}}}`
- Generated `cmd/<app>/main.go`: Construct the layered client once and inject into SDK client; pass to all commands

**A2 — Credential Resolution**
- Add `zalando/go-keyring` to `go.mod`
- `internal/cli/auth.go`: Complete generated `internal/auth/resolver.go` with 5-tier resolution chain
- Generate `<APP>_<SCHEME>_<TYPE>` env var names from registry metadata
- Wire resolver into `AuthTransport`

**A3 — Retry Wiring**
- `internal/sdk/retry_enhancer.go`: Ensure defaults (3 attempts, 1s initial, 30s max) applied when `RetryConfig` is nil in registry
- Wire `RetryTransport` into `HTTPClientFactory` below `AuthTransport`

**A4 — Timeout Wiring**
- Generated `cmd/<app>/main.go`: Apply `OperationMeta.Timeout` (or global default) to `http.Client.Timeout`
- Global timeout configurable via `<APP>_REQUEST_TIMEOUT` env / config key

### Phase B: Output & UX

**B1 — Table Output**
- `pkg/registry/types.go`: Add `Display bool` to `SchemaMeta` (set from `x-cliford-display: true`)
- `internal/openapi/extensions.go`: Parse `x-cliford-display` extension on response schema properties
- `internal/cli/output.go`: Complete table renderer using `text/tabwriter`; column selection logic (display-marked → all → `--fields` override)
- Add `--output` flag (table|json|detail) and `--fields` flag to all commands

**B2 — Verbose Flag**
- `internal/cli/flags.go`: Add `--verbose`/`-v` persistent flag on root command
- Generate `VerboseTransport` that wraps any `http.RoundTripper` and prints to stderr when enabled
- Implement redaction: `Authorization`, `X-Api-Key`, any header matching secret/token/key/password pattern

**B3 — Interactive Prompts**
- `internal/cli/prompts.go` (new): Generate `promptMissingArgs()` helper for each command
- Called at start of `RunE` when stdin is a TTY; iterates missing required params
- Skip prompt and print usage error when stdin is not TTY

**B4 — Confirm Prompt**
- Add `--yes`/`-y` flag to DELETE operations and operations with `x-cliford-confirm: true`
- Generate `confirmAction()` helper; show `"<message> [y/N]: "` prompt; default No

### Phase C: Pagination

**C1 — CLI Pagination Flags**
- `internal/sdk/pagination_enhancer.go`: Complete generation of `--page-token`, `--offset`, `--page`, `--limit` flags from `PaginationConfig`
- Add `--all` flag that drives `PageIterator` loop, accumulates results, outputs combined table/JSON

### Phase D: OAuth2 & Hooks

**D1 — OAuth2 Client Credentials**
- Add `golang.org/x/oauth2` to `go.mod`
- `internal/sdk/oauth_enhancer.go` (new): Generate `OAuthTokenSource` using `clientcredentials.Config` with in-memory cache (`ReuseTokenSource`)
- Store/retrieve tokens from keychain with TTL awareness; proactive refresh 60s before expiry
- Wire into `AuthTransport` when scheme type is `oauth2`

**D2 — Hooks**
- `internal/hooks/`: Complete shell hook executor (exec, JSON stdin, exit code handling)
- Add `hashicorp/go-plugin` to `go.mod`; complete go-plugin hook executor
- Wire hooks into generated `internal/client/factory.go` middleware chain
- Wire `FeaturesConfig.Hooks` toggle via Viper

### Phase E: Config, Distribution & Docs

- `internal/config/`: Complete generated `FeaturesConfig` struct; wire into Viper loading in generated `main.go`
